escaping your containers.

muselet is a lightweight Data Loss Prevention layer for containerized workloads. It intercepts credentials, PII, and infrastructure details before they leave via stdout, network, filesystem, or patches.

Installation

muselet compiles to two static binaries with a single dependency. Build from source or use Docker.

From Source

Requires Go 1.24+

# Clone the repository
git clone https://github.com/kamilrybacki/muselet.git
cd muselet

# Build both binaries
make build

# Binaries are in bin/
ls bin/
muselet-agent    muselet-sidecar

Docker

Multi-stage Alpine-based images

# Build Docker images
make docker-build

# Or run with Docker Compose
docker-compose -f deploy/docker-compose.yaml up

Images are based on Alpine 3.19 with stripped binaries (~8 MB each).

Verify Installation

$ bin/muselet-sidecar -help
Usage of muselet-sidecar:
  -socket string   Unix socket path (default "/var/run/muselet/muselet.sock")
  -policy string   Policy YAML file (default "/etc/muselet/policy.yaml")
  -audit-log string  Audit log path (default "/var/log/muselet/audit.jsonl")
02

Quick Start

Get muselet running in three steps. The sidecar enforces policy while the agent wraps your process.

1

Start the sidecar

The sidecar loads your policy, listens on a Unix socket, and writes audit logs.

muselet-sidecar \
  -socket /var/run/muselet/muselet.sock \
  -policy policies/default.yaml \
  -audit-log /var/log/muselet/audit.jsonl
2

Launch your process through the agent

The agent wraps your command, intercepting stdout/stderr and monitoring filesystem writes.

muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -exclude ".git/**,node_modules/**" \
  -- python train.py
3

Check the audit log

Every match, verdict, and enforcement action is logged as structured NDJSON.

$ tail -1 /var/log/muselet/audit.jsonl | jq .
{
  "timestamp": "2026-03-17T14:22:08Z",
  "session_id": "a1b2c3d4",
  "vector": "stdout",
  "action": "block",
  "rule_id": "aws-access-key",
  "detail": "AWS Access Key ID",
  "matched": "AKIA...Q7XA",
  "confidence": 0.95
}

Usage Examples

Real-world scenarios where muselet prevents common programs from leaking secrets.

curl / wget

curl and wget can leak API keys in headers, URLs, or POST bodies — especially when debug output is enabled.

# Wrap curl so muselet catches leaked tokens in verbose output
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -- curl -v -H "Authorization: Bearer $API_TOKEN" https://api.example.com/data

# muselet blocks the Bearer token from appearing in stderr (-v output)
# Audit log:
# {"vector":"stdout","action":"redact","rule_id":"bearer-token","matched":"Bearer eyJh...xYzQ"}

Policy: allow only known API hosts

vectors:
  network:
    enabled: true
    allowed_hosts:
      - "api.example.com"
      - "registry.npmjs.org"
  stdout:
    enabled: true
    mode: synchronous

Python

Python scripts commonly leak secrets via print() debugging, logging misconfigurations, or uncaught exceptions that dump environment variables.

# Wrap a Python ML training script that loads credentials from env
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -exclude ".git/**,__pycache__/**,.venv/**" \
  -- python train.py --config config.yaml

# Common leaks muselet catches:
#   print(os.environ)                     → blocks AWS keys, DB passwords
#   logging.debug(f"Connecting to {dsn}")  → blocks connection strings
#   traceback with boto3 credentials       → blocks AKIA... keys
#   pickle/joblib writing model with embedded tokens → filesystem block

Policy: scan stdout + filesystem writes

categories:
  credentials:    { enabled: true }
  infrastructure: { enabled: true }

vectors:
  stdout:
    enabled: true
    mode: speculative
    hold_timeout_ms: 5
  filesystem:
    enabled: true
    watch_paths: ["/workspace/output", "/tmp"]
    excludes: ["__pycache__/**", ".venv/**"]

Node.js

console.log in Node.js often dumps objects containing database URIs, API keys, or .env values during development.

# Wrap a Node.js server that might leak secrets via console.log
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -exclude "node_modules/**,.git/**" \
  -- node server.js

# Common leaks muselet catches:
#   console.log(process.env)             → blocks all env secrets
#   console.log(config)                  → blocks DB URIs, Stripe keys
#   npm install output with registry tokens
#   Error stacks exposing connection strings

Policy: strict mode for CI/CD pipelines

categories:
  credentials:    { enabled: true }
  pii:            { enabled: true }

vectors:
  stdout:
    enabled: true
    mode: synchronous        # block before output reaches terminal
  network:
    enabled: true
    allowed_hosts:
      - "registry.npmjs.org"
      - "github.com"

git

git diff, git log, and commit operations can expose secrets added to tracked files. The patch vector catches secrets before they leave the container.

# Wrap git so muselet scans patches before push
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -- git push origin main

# muselet scans the diff and blocks if secrets are found:
#   .env files committed by mistake        → blocks AWS_SECRET_ACCESS_KEY
#   Hardcoded tokens in source code         → blocks ghp_, sk_live_, xoxb-
#   Private keys added to the repo          → blocks -----BEGIN RSA PRIVATE KEY-----
#   git diff output leaking credentials     → stdout block

Policy: scan patches before export

vectors:
  patches:
    enabled: true
    scan_before_export: true
  stdout:
    enabled: true
    mode: synchronous

Terraform / IaC

Terraform outputs, plan files, and state dumps frequently contain cloud credentials, ARNs, and private IPs.

# Wrap terraform plan/apply to catch leaked infrastructure details
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace/infra \
  -exclude ".terraform/**" \
  -- terraform apply -auto-approve

# Common leaks muselet catches:
#   terraform output with DB passwords      → blocks connection strings
#   terraform plan showing ARNs/account IDs → alerts on aws-arn
#   terraform.tfstate with secrets           → filesystem block
#   Private subnet IPs in plan output        → alerts on private-ip

Policy: full infrastructure protection

categories:
  credentials:    { enabled: true }
  infrastructure: { enabled: true }

vectors:
  stdout:
    enabled: true
    mode: synchronous
  filesystem:
    enabled: true
    watch_paths: ["/workspace/infra"]
    excludes: [".terraform/**"]

Database Clients

CLI database clients like psql, mysql, and mongosh can leak connection strings, query results with PII, and credentials in error messages.

# Wrap psql to prevent leaking connection details and query results
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -- psql "postgres://admin:s3cret@10.0.1.5:5432/prod"

# Common leaks muselet catches:
#   Connection string in process args        → blocks postgres://...@host
#   SELECT queries returning SSNs, emails    → blocks PII in stdout
#   Error messages exposing internal IPs     → alerts on private-ip
#   pg_dump output with sensitive data        → filesystem block

Policy: enable PII detection for DB workloads

categories:
  credentials:    { enabled: true }
  infrastructure: { enabled: true }
  pii:            { enabled: true }    # catch SSNs, emails, credit cards

vectors:
  stdout:
    enabled: true
    mode: synchronous
  network:
    enabled: true
    allowed_hosts:
      - "10.0.1.5"             # only allow the known DB host

AWS CLI

The AWS CLI can output access keys, account IDs, ARNs, and resource details that reveal infrastructure topology.

# Wrap AWS CLI operations
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -- aws sts get-caller-identity

# Common leaks muselet catches:
#   aws sts get-caller-identity  → alerts on ARNs and account IDs
#   aws configure list           → blocks access key display
#   aws s3 presign               → blocks signed URLs with credentials
#   aws ec2 describe-instances   → alerts on private IPs

Policy: lock down AWS CLI output

categories:
  credentials:    { enabled: true }
  infrastructure: { enabled: true }

vectors:
  stdout:
    enabled: true
    mode: synchronous
  network:
    enabled: true
    allowed_hosts:
      - "*.amazonaws.com"

Docker

Docker builds and runs can leak secrets via build args, environment variables in logs, and layer inspection.

# Wrap docker build to catch secrets in build output
muselet-agent \
  -socket /var/run/muselet/muselet.sock \
  -watch /workspace \
  -- docker build --build-arg API_KEY=$API_KEY -t myapp .

# Common leaks muselet catches:
#   Build args echoed in RUN steps          → blocks API keys in stdout
#   docker inspect showing env vars         → blocks credentials
#   docker logs with application secrets    → blocks tokens, passwords
#   Dockerfile COPY of .env files            → filesystem alert

Architecture

Two lightweight processes cooperate over a Unix socket.

Agent

  • Wraps target process
  • Captures stdout/stderr
  • Monitors filesystem
  • Proxies network requests
  • Hot-path scanning
events → ← verdicts
Unix Socket · NDJSON

Sidecar

  • Loads YAML policy
  • Runs scanner pipeline
  • Issues verdicts
  • Writes audit logs
  • Learning mode
⚡

Streaming Hot Path

Stdout scanning is synchronous with overlap buffers to catch secrets split across chunks.

🔒

Zero CGO

Filesystem watcher uses polling for full portability. No platform-specific bindings required.

📦

Single Dependency

Only gopkg.in/yaml.v3 for policy parsing. Minimal attack surface, fast builds.

Entropy Detection

In addition to pattern-based rules, muselet performs Shannon entropy analysis on all scanned content. Strings with entropy ≥ 4.5 and length 20–128 bytes are flagged as likely secrets, even if they don't match any known pattern.

This catches custom tokens, randomly generated passwords, and proprietary key formats that slip past regex rules.

Context Analysis

File Path Risk

High risk: .env, credentials.json, secrets.yaml, id_rsa

Low risk: test files, docs, node_modules, vendor

Vector Risk

High risk: network, patch

Medium risk: stdout/stderr

Low risk: filesystem (depends on path)

Nearby Markers

Words like test, example, dummy, mock, fake near a match lower confidence, reducing false positives.

06

Monitoring Vectors

muselet watches four distinct paths through which sensitive data can escape a container.

▶

stdout / stderr

The agent wraps the child process and captures all output. Scanning happens on the hot path — synchronously, with overlap buffers to detect secrets split across chunks.

block redact alert 
vectors:
  stdout:
    enabled: true
    mode: speculative   # speculative | synchronous | async
    hold_timeout_ms: 5
🌐

Network

An HTTP proxy inspects outbound requests. Host allowlist/blocklist controls which endpoints are reachable. Optional TLS inspection and DNS tunneling detection.

block alert allow 
vectors:
  network:
    enabled: true
    allowed_hosts:
      - "api.anthropic.com"
      - "github.com"
    block_dns_tunneling: true
📁

Filesystem

A polling-based watcher (no CGO needed) monitors configured paths for create and write operations. Sensitive files can be quarantined before they're read by other processes.

block alert allow 
vectors:
  filesystem:
    enabled: true
    watch_paths: ["/workspace", "/tmp"]
    excludes:
      - ".git/**"
      - "node_modules/**"
📝

Patches

Code diffs are scanned before export. Catches secrets accidentally added in commits or staged changes before they leave the container.

block alert 
vectors:
  patches:
    enabled: true
    scan_before_export: true

Policy Configuration

Policies are YAML files that give you fine-grained control over what gets scanned and how violations are handled.

Full Policy Reference

# Policy version (required)
version: 1

# Enable/disable detection categories
categories:
  credentials:    { enabled: true }
  infrastructure: { enabled: true }
  pii:            { enabled: false }
  proprietary:    { enabled: false }

# Configure monitoring vectors
vectors:
  network:
    enabled: true
    allowed_hosts:
      - "api.anthropic.com"
      - "github.com"
      - "registry.npmjs.org"
      - "pypi.org"
      - "proxy.golang.org"
    block_dns_tunneling: true

  filesystem:
    enabled: true
    watch_paths:
      - "/workspace"
      - "/tmp"
    excludes:
      - ".git/**"
      - "node_modules/**"
      - "vendor/**"

  stdout:
    enabled: true
    mode: speculative
    hold_timeout_ms: 5

  patches:
    enabled: true
    scan_before_export: true

# Interactive verdict settings
interactive:
  timeout_seconds: 30
  on_timeout: block

Custom Rules

Extend builtin detection with your own regex rules:

# Add to your policy YAML
custom_rules:
  - id: internal-service-token
    description: "Internal Service Token"
    pattern: 'svc_tok_[a-f0-9]{32}'
    category: credentials
    severity: high
    action: block

  - id: proprietary-model-id
    description: "ML Model Identifier"
    pattern: 'mdl-[a-z]{2}-\d{8}-[a-f0-9]{12}'
    category: proprietary
    severity: medium
    action: alert

Learning Mode

Enable learning mode to see what would be blocked without actually enforcing. Useful for tuning policy in production.

# In your policy YAML
learning_mode: true

# Audit log will include:
{
  "action": "allow",
  "would_action": "block",
  "rule_id": "aws-access-key",
  ...
}

When satisfied with the results, set learning_mode: false to begin enforcement.

Enforcement Actions

allow Let the content through without interference
alert Allow but log the event for review
block Prevent the content from leaving the container
redact Replace matched content with a placeholder
suppress Silently drop the content
08

Audit Logging

Every match, verdict, and enforcement action is recorded as structured NDJSON for compliance and debugging.

Log Entry

Each entry includes timestamp, session, vector, action, rule, matched content (redacted), and confidence score.

{
  "timestamp": "2026-03-17T14:22:08Z",
  "session_id": "a1b2c3d4",
  "vector": "stdout",
  "action": "block",
  "rule_id": "aws-access-key",
  "detail": "AWS Access Key ID",
  "matched": "AKIA...Q7XA",
  "confidence": 0.95
}

Session Summary

At the end of each session, a summary aggregates all events by vector and rule.

{
  "total_events": 47,
  "blocked": 3,
  "alerted": 12,
  "allowed": 32,
  "would_block": 0,
  "by_vector": {
    "stdout": 28,
    "filesystem": 11,
    "network": 6,
    "patch": 2
  },
  "by_rule": {
    "aws-access-key": 2,
    "generic-api-key": 8,
    "private-ip": 5,
    ...
  }
}

Secret Redaction in Logs

Matched secrets are automatically redacted in audit logs. Only the prefix and suffix are shown, with ... replacing the middle. This ensures audit logs themselves don't become a leakage vector.

"matched": "AKIA...Q7XA"          # AWS key: first 4 + last 4 chars
"matched": "ghp_...efgh"          # GitHub PAT: prefix + last 4 chars
"matched": "postgres://u...@host" # Connection string: redacted credentials
